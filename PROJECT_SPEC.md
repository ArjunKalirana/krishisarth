## 1. Project overview

KrishiSarth is a closed-loop, AI-driven Smart Water and Nutrient Management System for Indian smallholder and mid-scale farmers. It shifts irrigation from fixed schedules to predictive, crop-centric decisions by combining long-range IoT hardware (LoRa mesh) with cloud-based machine learning.

### Core value proposition

| Problem | KrishiSarth solution |
|---------|---------------------|
| Water wasted on uniform irrigation | Zone-by-zone AI scheduling based on real soil data |
| Nutrient leaching from over-irrigation | EC-monitored fertigation with precision dosing |
| Farmers need to be on-site to operate pumps | Mobile dashboard + remote control from anywhere |
| No visibility into soil health | Three-depth soil monitoring every 15 minutes |
| Expensive to run pumps unnecessarily | Rain-aware AI cancels irrigation before it's needed |

## 2. System architecture

### 2.1 Component overview

```
Field Layer        →    Edge Layer       →    Cloud Layer       →    Client Layer
────────────────        ─────────────         ─────────────          ─────────────
ESP32 sensor nodes      Raspberry Pi          FastAPI backend        Web dashboard
(soil · temp · EC)  →  gateway          →    PostgreSQL             Mobile PWA
LoRa mesh network       MQTT publish          InfluxDB
Pump / valve relay      Local SQLite          Redis + Celery
                        buffer                AI engine
                                              MQTT broker
```

### 2.2 Data flows

**Sensor → Cloud (every 15 min):**
ESP32 reads sensors → LoRa transmit → Pi aggregates → MQTT publish → Ingestion service → InfluxDB write

**AI decision cycle (every 15 min via Celery Beat):**
Fetch sensor data from InfluxDB → Fetch weather forecast → Run LSTM + RF inference → Save decision → If confidence ≥ 60% and action = irrigate → Enqueue irrigation Celery task

**Dashboard (live):**
Browser WebSocket connection → FastAPI WS handler → Publishes InfluxDB latest + Redis cache → Client renders cards

## 3. Hardware specification

### 3.1 Field sensor node (ESP32)

| Component | Specification |
|-----------|-------------|
| MCU | ESP32-WROOM-32 |
| Radio | SX1278 LoRa (915 MHz, up to 10 km range) |
| Soil moisture | Capacitive probe, 3-depth (0-15 cm, 15-30 cm, 30-60 cm) |
| Temperature | DHT22 (±0.5°C accuracy) |
| EC sensor | Electrical conductivity probe, 0–10 dS/m |
| pH sensor | Analog probe with calibration offset |
| Solar | 2W panel + 3.7V LiPo battery |
| Power draw | ~80 mA active, ~10 µA deep sleep |
| Sleep cycle | Wake every 15 min, sample, transmit, sleep |

### 3.2 Edge gateway (Raspberry Pi 4)

- Runs Mosquitto MQTT broker (local) and Paho MQTT client (cloud publisher)
- SQLite buffer: stores up to 24h of readings for offline resilience
- Relay module: controls pump and valve on/off signals
- Hardware safety timer: cuts relay after max(scheduled_duration + 10 min) regardless of software state
- OTA firmware updates for ESP32 nodes via SPIFFS filesystem

## 4. Database design

### 4.1 PostgreSQL schema (relational data)

**Tables:** farmers, farms, zones, devices, irrigation_schedules, ai_decisions, fertigation_logs, alerts

**Key relationships:**
- farmers 1→N farms → 1→N zones → 1→N devices
- zones 1→N irrigation_schedules, ai_decisions, fertigation_logs
- farms 1→N alerts (zone_id nullable for farm-level alerts)

**Key constraints:**
- All IDs: UUID generated by `gen_random_uuid()`
- Cascading deletes: child records deleted with parent
- Partial indexes on high-frequency queries (e.g., pending schedules, unread alerts)

### 4.2 InfluxDB schema (time-series sensor data)

**Bucket:** krishisarth_sensors | Retention: 90 days | Downsampling: 7d → 1h, 30d → 1d

**Measurements:**
- `soil_readings` — moisture, temp, EC, pH (tags: zone_id, device_id, depth)
- `ambient_readings` — temp, humidity, solar intensity (tags: zone_id, device_id)
- `water_quality` — pH, EC, turbidity, nitrate (tags: farm_id, device_id)
- `pump_telemetry` — flow, pressure, is_running, power (tags: zone_id, device_id)

## 5. AI / ML specification

### 5.1 Models

**LSTM (moisture prediction):**
- Input features: soil moisture (3 depths), temperature, humidity, solar intensity, last irrigation timestamp, crop stage encoding
- Output: predicted moisture at 6h, 12h, 24h intervals
- Training data: 90 days × all zones × 15-min intervals
- Framework: Keras · TensorFlow

**Random Forest (irrigation decision):**
- Input features: current moisture, LSTM predictions, rain probability, EC level, crop stage, days since last irrigation, tank level
- Output: {decision: irrigate|skip|supplement_nutrient|alert, confidence: 0–1}
- Framework: scikit-learn

### 5.2 Decision thresholds

| Confidence | Action |
|-----------|--------|
| ≥ 0.80 | Auto-execute |
| 0.60 – 0.79 | Save decision, flag for manual review |
| < 0.60 | Save as informational only, no action |

### 5.3 Fallback chain

1. Full LSTM + weather forecast
2. LSTM only (weather API unavailable — treat rain probability as 0%)
3. Rule-based threshold: if moisture < 25%, irrigate for 20 min
4. Fixed time schedule: irrigate at 6:00 AM daily

## 6. API specification

### 6.1 Conventions

- Base URL: `https://api.krishisarth.in/v1`
- All requests/responses: `application/json`
- Authentication: `Authorization: Bearer <jwt_token>`
- Error envelope: `{"success": false, "error": {"code": "...", "message": "...", "request_id": "..."}}`

### 6.2 Endpoint index

| Method | Path | Description |
|--------|------|-------------|
| POST | /auth/register | Register farmer |
| POST | /auth/login | Login, receive tokens |
| POST | /auth/refresh | Rotate access token |
| GET | /farms | List farmer's farms |
| POST | /farms | Create farm |
| GET | /farms/:id | Farm details + zones |
| POST | /farms/:id/zones | Create zone |
| PATCH | /zones/:id | Update crop stage / active |
| GET | /farms/:id/dashboard | Full live snapshot |

| POST | /zones/:id/stop | Stop irrigation |
| POST | /zones/:id/fertigation | Inject nutrients |
| GET | /zones/:id/ai-decisions | Decision history |
| POST | /zones/:id/ai-decisions/run | Trigger on-demand AI |

| GET | /farms/:id/alerts | Alert list |
| PATCH | /alerts/:id/read | Mark alert read |

### 6.3 Authentication flow

Access token: JWT HS256, 24h expiry, stored in JS memory (never localStorage)
Refresh token: JWT HS256, 30d expiry, stored in httpOnly Secure SameSite=Strict cookie
Refresh token rotation: each refresh revokes old jti in Redis, issues new jti
Device auth: SHA-256 hashed API key in X-Device-Key header (IoT hardware only)

## 7. Security requirements

- All traffic: HTTPS / WSS only (TLS 1.3)
- Passwords: bcrypt, cost factor 12
- JWT: HS256 with secrets ≥ 256 bits, stored in environment variables
- Refresh token replay detection: jti revocation in Redis
- Cross-tenant protection: ownership middleware on every resource endpoint — always 403, never 404 for unauthorized access
- Rate limiting: 100 requests/min per token (Redis counter)
- Brute force protection: lock account for 15 min after 5 failed logins
- Device key storage: SHA-256 hash in DB, plaintext only at provisioning time

## 8. Error handling policy

**Hardware failures:** Pi relay safety timer is the last line of defense — cuts pump power independent of all software.

**Sensor anomalies:** Readings outside physical bounds are rejected and not written to InfluxDB. Generates SENSOR_FAULT alert.

**AI engine degraded:** Four-level fallback chain (see Section 5.3). Dashboard shows current mode.

**Network partitions:** Pi gateway buffers 24h locally. Frontend retries with exponential backoff. Dashboard falls back to cached data.

**Database failures:** Dashboard data cached in Redis (60s TTL). API returns 503 with cache notice.
